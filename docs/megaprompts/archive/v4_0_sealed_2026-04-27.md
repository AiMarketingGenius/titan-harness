## 3. ARCHITECTURE — 4 LAYERS

### Layer 1 — Context-fill auto-compact
- **Trigger:** Agent context usage ≥ 70% of model window
- **Detection:** Each agent emits `context_pct` estimate per task completion to `op_agent_health` (Supabase)
- **Action:**
  1. Snapshot current state: `log_decision(tag="pre_compact:<agent>:<ts>", text=<state_summary>)`
  2. SIGTERM agent at next task boundary
  3. systemd auto-restart
  4. `cold_start_hydrate` runs with last 5 decisions + condensed summary of pre-compact state
- **Excluded from Layer 1:** Mercury, factory_status_writer, persona_summary_daemon (stateless or single-purpose)

### Layer 2 — Error-pattern force-reset (HIGHEST ROI — BUILD FIRST)
- **Trigger:** ANY of these within rolling 30-min window per agent:
  - 2+ entries with tag matching `aletheia-violation` or `aletheia:false_completion`
  - 3+ consecutive tool-call failures
  - 3+ identical error-retry attempts
- **Detection:** `aletheia_pattern_detector.py` polling `op_decisions` every 5 min
- **Action:**
  1. Write kill signal to `/opt/amg-governance/agent_kill_queue/<agent>.signal`
  2. Agent's main loop checks for kill signal at task boundary
  3. Agent: snapshot in-flight state → `log_decision(tag="drift_reset:<agent>:<reason>")` → SIGTERM self
  4. systemd auto-restart → `cold_start_hydrate` runs → fresh context

### Layer 3 — Time-ceiling soft warning
- **Trigger:** 4 hours (14400s) continuous uptime per agent
- **Action:**
  - `log_decision(tag="time_ceiling_warning:<agent>")` — visible in Atlas Recent Decisions panel
  - Auto-snapshot current state
  - **No automatic restart** — Solon decides
- **Override:** Agent continues unless Solon writes `SOLON_OVERRIDE.flag` in governance dir

### Layer 4 — Baseline behavior verification (canary)
- **Trigger:** Every 6 hours per agent type
- **Method:** Cerberus issues a known-correct read-only canary task; compares output hash to expected
- **Action:** If 3 consecutive canaries deviate → `log_decision(tag="drift_baseline_breach:<agent>")` → trigger Layer 2 reset
- **Build incrementally:** Start with 3 agent types (Titan, daedalus-hercules, hephaestus-hercules), expand after first week

### Layer 5 — Mandatory scheduled restart (ENHANCED IN v2.0)
- **Trigger:** Every 4 hours per agent (rolling cadence, agent-specific stagger to avoid mass-restart spike)
- **Purpose:** Bounded uptime regardless of detected drift. Prevents slow-creep degradation that doesn't trip Layers 1–4.
- **CORE PRINCIPLE (v2.0):** Restarts NEVER fire mid-flight. They wait for safe boundaries and respect criticality locks.

#### Hierarchical task-boundary respect (v2.0)

The orchestrator must distinguish three boundary types and only restart at the safest available:

| Boundary type | Definition | Restart-safe? |
|---|---|---|
| **Atomic primitive** | Single tool call, single API request, single SQL transaction | ❌ NEVER kill mid-call. Wait. |
| **Sub-task complete** | One step of a multi-step task finished, next step not yet started | ✅ Safe — preferred kill point |
| **Top-level task complete** | Entire task done, agent idle awaiting next dispatch | ✅✅ Optimal kill point |
| **Conversational turn** (chiefs only) | User message → full response → idle | ✅ Safe with 60s save window |

Agent self-reports current boundary state in `op_agent_health.current_boundary` (updated per tick). Orchestrator NEVER kills with `current_boundary='atomic_primitive'`.

#### Criticality lock (v2.0)

Any agent can mark itself "in critical work" — restart orchestrator defers until released:

- **Lock mechanism:** Agent writes `/opt/amg-governance/critical_lock/<agent>.lock` containing:
  ```
  reason=<short description>
  acquired_at=<ISO timestamp>
  ttl_seconds=<int, max 1800 = 30 min>
  expected_release=<ISO timestamp>
  ```
- **Auto-expiry:** TTL hard-capped at 30 min. Agent must re-acquire (renews) if more time needed.
- **Hard ceiling:** Even with lock, if total uptime exceeds 12 hours, force restart anyway. Drift accumulates regardless of intent.
- **What qualifies as critical:**
  - Multi-step deployment in progress
  - Mid-pipeline data transformation that can't checkpoint
  - Active client communication thread (chiefs only)
  - Verification gate execution (Aletheia)
  - In-flight Mercury primitive chain
- **Solon override:** `/opt/amg-governance/SOLON_NEVER_RESTART/<agent>.flag` — soft semaphore, agent stays alive until flag removed (use sparingly; bypasses 12hr hard ceiling only with explicit Solon ack)

#### Restart action by agent class

- **Hard-reset class** (Titan, Achilles, builders, Mercury, Aletheia, Warden, Cerberus, factory_status_writer, persona_summary_daemon):
  1. Check criticality lock → defer if held
  2. Wait for `current_boundary != 'atomic_primitive'`
  3. Snapshot state via `log_decision(tag="scheduled_rest:<agent>:<ts>", text=<full state narrative>)`
  4. SIGTERM
  5. systemd auto-restart + Universal Bootstrap (§3B)
- **Soft-handoff class** (Hercules, Nestor, Alexander):
  1. Check criticality lock → defer if held
  2. Inject "⏰ Scheduled rest in 60s — wrap any in-flight thought" into chief's chat UI
  3. Wait 60s OR until next conversational turn boundary, whichever is sooner
  4. Snapshot conversational state + working context to `log_decision`
  5. Close chat session
  6. Solon's `/Applications/*.app` shows "Restarting..." for ~5s
  7. Universal Bootstrap (§3B) brings chief back with full memory + last conversation summary
  8. Solon picks up where he left off (memory persists; only working context resets)

#### Cascade-restart throttle (v2.0)

To prevent MCP overload when multiple agents wake simultaneously:
- Orchestrator enforces max 2 concurrent restart-in-progress
- Stagger schedule (below) is staggered by ≥30 min for major agents to avoid collision
- Bootstrap MCP queries are batched + cached for 60s after a restart wave

#### Stagger schedule (avoid simultaneous restarts)

|---|---|
| Titan | 00:00, 04:00, 08:00, 12:00, 16:00, 20:00 |
| Achilles | 00:30, 04:30, 08:30, 12:30, 16:30, 20:30 |
| Hercules | 01:00, 05:00, 09:00, 13:00, 17:00, 21:00 |
| Nestor | 01:30, 05:30, 09:30, 13:30, 17:30, 21:30 |

- **Skip override:** `/opt/amg-governance/no_scheduled_rest/<agent>.flag` skips the next scheduled restart (one-shot, auto-expires)

## 3B. UNIVERSAL BOOTSTRAP PROTOCOL (NEW IN v2.0)

**Every agent — not just chiefs — must hydrate from MCP on cold-start.** Currently `cold_start_hydrate` runs only on chief launch. Mercury, Aletheia, Warden, Cerberus, the 12 builders, factory_status_writer, persona_summary_daemon — most have no formal bootstrap. After Layer 5 forces a restart, these agents come back blank. Unacceptable.

### Required bootstrap sequence (all agents)

When any in-scope agent starts (cold-start OR Layer 5 scheduled restart OR Layer 2 forced reset OR shift handoff), it MUST run `universal_bootstrap.py <agent_id>` BEFORE accepting work:

| Step | Action | Source | Failure mode |
|---|---|---|---|
| **0** | **Load lcache (NEW v2.1)** — read `/opt/amg-governance/agent_local_state/<agent>.lcache` if present and ≤30min old. Provides "what I was just doing" continuity. | Local disk (3KB max) | Continue without (cold-bootstrap fallback) |
| 1 | **Identity load** | Static config: agent_id, role, owner, host | Fail-fast → systemd retry |
| 2 | **Standing rules load** | `get_bootstrap_context(scope="<agent_class>")` — P10 rules relevant to agent | Retry 3x, then alert |
| 3 | **Sprint state load** | `get_sprint_state(project_id="<owner_project>")` | Continue with last cached |
| 4 | **Recent decisions load** | `get_recent_decisions(count=15, project_filter="<scope>")` | Continue with empty |
| 5 | **Active task load** | Query `op_task_queue WHERE assigned_to=<agent> AND status IN ('queued','in_progress')` | Continue with empty |
| 6 | **Cross-agent context load** | `search_memory("<agent_owner> recent activity", count=5)` | Continue with empty |
| 7 | **Awakeness self-check** | Verify steps 0–6 above; emit health beacon | If 3+ fail → refuse work, alert |

**Step 0 + Steps 1–6 = total memory.** MCP gives world truth. lcache gives personal continuity. Together: agent wakes up knowing both "what's happening" and "what I was doing."

### Awakeness self-check (mandatory)

After steps 1–6, agent confirms it actually has memory by running 3 read-back tests:

1. **Identity confirm:** "Who am I? What's my owner?" → must match config
2. **Sprint awareness:** "What's the current sprint and my role in kill_chain?" → must return non-empty
3. **Recent context:** "What was my last logged decision?" → must return real entry, not "I don't know"

If any of these fails, agent writes `log_decision(tag="bootstrap_failure:<agent>:<step>")`, refuses to accept new work, and triggers alert. Cerberus monitors for these.

### Bootstrap caching for cascade-restart safety

When multiple agents bootstrap within 60s (Layer 5 stagger spike, or recovery from outage):
- First agent's MCP calls populate a process-local cache at `/opt/amg-governance/bootstrap_cache/`
- Subsequent agents within 60s read from cache where possible
- Cache invalidates after 60s
- Reduces MCP load 10x during restart waves

## 3C. COLLECTIVE MEMORY PROJECT (NEW IN v2.0 — "Manhattan Project for Memory")

**The vision:** every agent contributes to ONE unified, complete, ever-growing persistent memory. Not silos. Not isolated logs. A coherent system memory that any agent can query and that grows richer with every action.

**The principle:** if it happened in the AMG factory, the memory remembers. Solon never has to re-explain anything. Agents never re-litigate decisions. The system gets smarter every day.

### Five mechanisms (build all in Phase 8)

#### Mechanism 1 — Universal decision logging discipline

Every agent (not just chiefs) logs every meaningful action via `log_decision`:
- **Action types that MUST be logged:**
  - Any deliverable produced
  - Any decision/choice made between alternatives
  - Any error encountered + how resolved
  - Any blocker hit
  - Any new learning ("I tried X, it failed because Y")
  - Any cross-agent interaction
- **Anti-pattern:** silent action. If agent does it, MCP knows.
- **Rate:** target ≥1 decision per task completion per agent
- **Audit:** Cerberus monitors `log_decision` rate per agent — agents below threshold flagged for inspection

#### Mechanism 2 — Cross-agent visibility

Currently agents mostly see their own decisions. v2.0 changes this:
- Every agent's bootstrap (§3B step 6) includes `search_memory("<my_owner> recent activity")` — sees siblings' work
- New MCP tool: `get_cross_agent_context(my_agent, scope='task'|'owner'|'global')` — returns curated cross-agent feed
- Atlas UI shows live cross-agent activity stream

#### Mechanism 3 — KB graduation pipeline

`op_decisions` is ephemeral (tactical). `op_memory_vectors` (KB chunks) is permanent (strategic). Important decisions need to graduate from one to the other:

- **Trigger:** Agent flags decision with tag `kb_candidate` when logging
- **Review:** Daily Hephaestus pass surfaces all `kb_candidate` entries
- **Promotion:** Aletheia or Solon approves → entry chunked, embedded, ingested into appropriate KB namespace (`kb:<owner>:<topic>`)
- **Result:** important learnings become retrievable by future agents at sub-second latency via embedding search

#### Mechanism 4 — Memory health metrics + coverage map

- **Per-namespace dashboards:**
  - Total chunks
  - Last write timestamp (freshness)
  - Retrieval frequency (how often queried)
  - Hit rate (how often retrieval surfaces relevant content)
  - Identified gaps (queries that returned nothing useful)
- **Coverage map:** visual representation of memory completeness by topic area, owner, time window
- **Solon UI:** Atlas panel "Memory Health" — see gaps, freshness, problem areas at a glance

#### Mechanism 5 — Memory feedback loop (supersede, never destroy)

When an agent retrieves memory and finds it stale, wrong, or superseded:
- Agent does NOT delete (audit trail sacred)
- Agent writes `log_decision(tag="memory_correction", text=<correction>, supersedes=<old_decision_id>)`
- `op_search_memory` RPC filters superseded entries from default queries (already implemented; v2.0 just makes it consistent)
- Aletheia tracks correction patterns — flags agents that frequently correct themselves (drift signal)

### Solon's role — Memory Editor

Solon gets a UI (Atlas panel) to:
- Review queued `kb_candidate` entries
- Approve/reject KB graduation
- Manually flag entries as superseded
- Add manual KB entries directly (bypass agent flow when needed)
- See memory health at a glance

### Phase 8 builds the foundation; full implementation may need separate megaprompt

Phase 8 (below) ships:
- Mechanism 1 (logging discipline + Cerberus audit)
- Mechanism 5 (feedback loop with supersede pattern)
- Foundation for Mechanisms 2–4 (cross-agent tool, graduation pipeline schema, health metrics table)

Full Mechanisms 2, 3, 4 polish + Atlas UI panels likely require a follow-up dispatch after Phase 8 ships. Don't let perfect block good — ship the foundation, iterate.

## 3D. LOCAL CONTINUITY CACHE — "lcache" (NEW IN v2.1)

**The gap:** Bootstrap from MCP loads global truth ("what's happening in the world") but doesn't capture the agent's last in-flight thought ("I was about to send the deployment command"). When an agent restarts, it knows the world but doesn't remember its own immediate intent.

**The fix:** Each agent maintains a tiny 3KB local cache file — its personal handoff buffer — that captures the absolute essentials of its working state pre-shutdown.

### Spec

| Field | Value |
|---|---|
| **Path** | `/opt/amg-governance/agent_local_state/<agent_id>.lcache` |
| **Size limit** | 3072 bytes (3 KB) — HARD ceiling, enforced at write time |
| **Format** | Compact YAML (or compressed JSON) |
| **Lifetime** | Valid for 30 minutes after last write — older = stale, ignored on read |
| **Owner** | Agent itself (read+write); root owns directory |
| **Encryption** | At-rest disk encryption only (filesystem level) — no per-file crypto |

### What goes in lcache (forced brevity = forced clarity)

```yaml
agent: daedalus-hercules
written_at: 2026-04-27T15:42:33Z
last_active_thought: "About to dispatch CT-0427-44 to artisan; awaiting kb_search result"
in_flight_task_id: CT-0427-44
in_flight_substep: 3 of 7
key_context_refs:
  - decision_id: 8f3a-...-c2
  - decision_id: 91b4-...-d7
  - kb_chunk_id: kb:hercules:doctrine#chunk_492
last_user_directive: "Solon said prioritize SHOP UNIS work over outbound"
intent_on_wake: "resume dispatch of CT-0427-44 substep 3 with kb_search context"
notes: "If kb_search returned >0 hits, send to artisan; if 0, escalate to Solon"
```

**Forced compression principle:** 3KB ≈ ~750 tokens ≈ ~500 words. Agents must distill, not dump. This mirrors what humans do with sticky notes — short, distilled, the absolute essentials. Anything you couldn't write on a small index card doesn't belong here.

### Write triggers

Agent writes to lcache:
- Before any planned shutdown (Layer 5 scheduled rest, Layer 2 forced reset, shift handoff)
- Every 5 minutes during active work (rolling refresh — survives unplanned crashes)
- After completing each sub-step of a multi-step task

### Read triggers

Agent reads from lcache during bootstrap step 0 (per §3B):
- If file exists AND `written_at` ≤ 30min ago → load and use
- If file missing OR stale → cold bootstrap (no continuity, but agent still wakes with full MCP context)

### Read-back validation

On load, agent verifies:
1. `agent_id` field matches its own identity (corruption check)
2. `written_at` is real ISO timestamp (sanity check)
3. Total file size ≤ 3072 bytes (truncation if violated)
4. No injection patterns (per AMG security scan)

### Why 3KB and not 30KB or 300KB?

- 3KB forces signal over noise. Agent must choose what truly matters.
- Loads in milliseconds (no I/O penalty)
- Negligible disk footprint × 20 agents = 60KB total cache layer
- Aligns with how human short-term memory works — bounded, focused, summary-form

### Cleanup

- Janitor (Hephaestus) sweeps lcache directory daily
- Files >24hr old archived to cold storage (in case of forensic need)
- Files corrupt/oversized auto-quarantined and replaced with fresh empty stub

## 3F. HYGEIA — CLEANING AGENT (NEW IN v3.0)

**Greek goddess of health, cleanliness, and preventative medicine. Daughter of Asclepius. Embodies hygiene and the active maintenance of wellbeing — not just curing sickness, but preventing it.**

Naming fits the AMG mythology pattern (Aletheia, Cerberus, Achilles, Hercules, Nestor, Alexander, Athena, Hephaestus, Mercury). She is the agent we don't have yet but desperately need.

### The gap she fills

Today, when an over-tired agent drifts and writes a false claim into MCP:
- **Aletheia** catches it (after the fact, via verification check)
- **Warden** evicts stuck locks (operational hygiene)
- ...and that's it. Nobody actively scrubs the poisoned data, audits affected KB chunks, notifies downstream agents, or detects drift before it produces violations.

Today's e3b4e92 cascade is exhibit A. Three Aletheia violations within 10 minutes. No agent took action to:
1. Mark Titan as a drift candidate after violation #1
2. Quarantine any KB chunks ingested from Titan's session that day
3. Notify other agents that Titan's recent claims should be re-verified
4. Prevent Titan from compounding the error 8 minutes later (he re-asserted the same false claim)

Hygeia owns this entire surface area.

### Hygeia's six responsibilities

#### 1. Continuous poison scanning (every 15 min)

Scan `op_decisions` for poison patterns:
- Aletheia violations from agents that worked >5 hours (per §3E shifts, this should be impossible — but if the system fails, Hygeia catches it)
- Decisions that contradict newer authoritative entries (factual drift)
- Tasks claiming completion but with no verifiable artifacts (extends Aletheia's pattern retroactively)
- Decisions referencing tool/path/file names that no longer exist (stale references)
- Heartbeats from agents past their shift end (zombie processes)

#### 2. Active scrubbing (mark, never delete)

When Hygeia identifies poison:
- Write `log_decision(tag="hygeia:scrubbed", supersedes=<poisoned_decision_id>, text=<correction reasoning>)`
- Update `op_decisions.void=true` on the poisoned entry (filtered from default queries; preserved for audit)
- Never `DELETE FROM op_decisions` — audit trail is sacred
- Trigger downstream notification (mechanism 4)

#### 3. KB chunk hygiene (daily audit)

`op_memory_vectors` chunks ingested from drifted-agent sessions could be poisoned KB. Hygeia:
- Cross-references each chunk's `source_decision_id` against the void/scrubbed list
- Quarantines suspect chunks (sets `quarantined=true`, removed from default retrieval)
- Flags for re-validation: either Solon manually approves, or original source is re-verified
- Never deletes chunks — audit trail principle applies to KB too

#### 4. Cross-agent notification (when poison is found)

When Hygeia scrubs a poisoned decision:
- Identify all agents that retrieved that decision in last 24hr (via `op_decisions_retrieval_log` if it exists; build it if not)
- Write `log_decision(tag="hygeia:correction_notice", text=<what was poisoned, what's now correct>)` scoped to those agents
- On their next bootstrap, those agents see the correction in step 4 (recent decisions) — no propagation of bad data

#### 5. Preventative drift detection (early warning)

Don't wait for Aletheia to catch a violation. Detect drift signals earlier:
- Agent retry rate climbing (3+ retries on same task in shift = drift signal)
- Agent's lcache `last_active_thought` repeating across shifts (looping)
- Agent's `context_pct` consistently >85% before scheduled restart (insufficient context budget)
- Agent's tool-call latency increasing over a shift (degradation)
- Agent's decision rate abnormally high or low vs baseline

When 2+ signals fire for an agent in 30 min: `log_decision(tag="hygeia:drift_warning:<agent>", severity=preemptive)` → Cerberus alerts + Layer 2 reset triggered

#### 6. Daily health report (preventative maintenance)

Every 24hr, Hygeia writes:
- `log_decision(tag="hygeia:daily_health_report", project_source="EOM")` containing:
  - Number of poison entries scrubbed (last 24hr)
  - Patterns detected (e.g., "Titan-A had 3 false-completion claims in shift A; recommend tighter context budget")
  - KB chunks quarantined
  - Agent quality scores (errors, retries, drift signals per agent)
  - Recommended preventative actions (e.g., "consider reducing daedalus-hercules max context to 60% of window — currently averaging 78%")
- Solon sees this in Atlas → Memory Health panel

### Hygeia's authority and limits

- **CAN:** Mark `void=true`, quarantine KB chunks, notify agents, trigger Layer 2 resets via Cerberus, write `log_decision`, recommend Solon actions
- **CANNOT:** Delete any record (only mark/quarantine), modify Standing Rules (P10s), restart Tier-2 singletons (must request via Solon), bypass Solon for Tier-1 destructive ops
- **Aletheia/Hygeia mutual check:** Aletheia verifies Hygeia's scrubs; Hygeia audits Aletheia's verifications. Atlas Watchdog doctrine in action.

### Hygeia runs on 4-shift rotation (per §3E)

Hygeia is herself prone to drift. 4 instances (Hygeia-A, -B, -C, -D), 6hr shifts, lcache continuity. If she cleans aggressively while drifted, she could "clean" valid data. Mutual check with Aletheia is the safety.

### Implementation footprint

- Single Python service: `hygeia.py` running as systemd unit (4 instances, shift rotation)
- Reads/writes to MCP via standard tools
- Adds 3 new MCP tools:
  - `mark_decision_void(decision_id, reason, scrubbed_by)` — supersede mechanism
  - `quarantine_kb_chunk(chunk_id, reason)` — KB hygiene
  - `flag_drift_signal(agent_id, signal_type, severity)` — preventative detection
- Cost: ~$3/mo Kimi-based (one Hygeia active at a time per shift schedule)
