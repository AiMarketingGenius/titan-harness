## CRITICAL defects (P0 — extension unshippable until fixed)
- **[service-worker.js:0]** No `chrome.runtime.onInstalled` or `chrome.runtime.onStartup` listeners.  
  **Description:** Service worker global arrays (`hotCache`, `pendingExtractions`) are lost on termination (MV3 idle timeout). In-flight captures stored only in `pendingExtractions` (in-memory) are not persisted to `chrome.storage.local` before idle shutdown, causing data loss.  
  **Fix:** Add `chrome.runtime.onInstalled` to load persistent state; on every `push` to `pendingExtractions`, immediately flush to `chrome.storage.local` (or a combined queue). Use `chrome.runtime.onSuspend` (if available) to dump in-memory state.

- **[service-worker.js:0]** Missing Einstein backend round-trip to `memory.aimarketinggenius.io`.  
  **Description:** No API call, auth header, or retry logic targets the required Einstein endpoint. The entire Einstein fact-checking pipeline appears to rely on Supabase (consumer_memories) only, not the designated backend.  
  **Fix:** Implement `fetch` to `memory.aimarketinggenius.io` with proper token/auth (e.g., bearer from storage), exponential backoff, and queueing. Wire the `runEinsteinFactCheck` function to this endpoint.

- **[chatgpt.js, claude.js, gemini.js, grok.js, perplexity.js, copilot.js]** Content scripts do **not** re‑attach `MutationObserver` on SPA navigation.  
  **Description:** All six platform scripts call `observeConversation` once when `platformInfo` is first available. If the host SPA replaces the conversation container (e.g., clicking a new thread link without full page reload), the observer becomes detatched and new assistant messages are silently missed.  
  **Fix:** In each script, listen for `THREAD_CHANGED` messages from `platform-detector.js` and `window.__AIMEMORY_PLATFORM` updates. On thread change, disconnect the previous observer and call `observeConversation` again with the updated `platformInfo`. Also reset `processedMessages` Set per thread to avoid false deduplication.

  **Description:** The platform detector lists `meta` as a supported platform, but no content script bundle is injected on those domains. If a user visits Meta AI, captures cannot occur.  
  **Fix:** Either add a `content_scripts` entry for Meta (if it is a target platform) or remove it from `host_permissions` and platform rules to avoid confusion.

- **[service-worker.js (suspect missing timer)]** Batch processing for `pendingExtractions` may never execute.

  **Fix:** Add a `setInterval` or use `chrome.alarms` to periodically drain `pendingExtractions`, ensuring the batch persists to storage before service‑worker death.

- **[popup.js:?]** `loadStats()` references `swStats.hotCacheSize` and other fields without confirmation that service worker’s `GET_STATS` returns them.  
  **Description:** If `service-worker.js`’s `getStats()` handler returns an object lacking these keys, `stat-memories` defaults to `0`, masking real counts.  
  **Fix:** Standardise the `GET_STATS` response shape and add fallback `0` on the popup side with `?? 0`.

- **[ui/thread-health.js:?]** Uses `chrome.storage.session` to persist depth‑fired state, but this API may be unavailable in older browsers (Chrome < 102).  
  **Description:** While Chrome 147 supports it, any enterprise environments with forced older versions will throw an error.  
  **Fix:** Gracefully degrade: catch `chrome.runtime.lastError` and fall back to `window.__aimg_depth_*_shown` flags.
